# 减肥游戏产品落地方案（App 版）

> 版本：V1.0（2026.5.22）
> 类型：Unity 原生 App 体感拳击对战游戏
> 首发平台：Android App
> 后续平台：iOS App
> 文档性质：产品落地交付方案（App 技术架构版）
> 关联文档：`docs/减肥游戏产品落地方案_Unity版.md`、`docs/减肥游戏-项目进度报告.md`

---

## 一、项目概览

### 1.1 方案定位

本方案是原微信小游戏方案的 App 版本，不覆盖原方案。核心调整不是放弃 Unity，而是将当前的 **Unity WebGL + 微信小游戏运行环境** 改为 **Unity 原生 App + 原生摄像头 + 端侧姿态识别**。

App 版本重点解决当前小游戏路线中的技术限制：

| 当前小游戏痛点 | App 版调整方向 |
|---|---|
| WebGL 运行环境限制 Unity 渲染、内存、线程和插件能力 | 改为 Unity Android/iOS 原生构建，直接使用移动端运行能力 |
| 姿态识别依赖 `wx.createVKSession`，机型覆盖不稳定 | 改为端侧 MediaPipe / MoveNet 姿态识别 |
| 服务端云推理延迟高，实时动作反馈慢 | 摄像头帧在手机本地推理，Unity 本地即时触发动作 |
| 包体受微信小游戏主包和 CDN 限制 | App 首包容量更宽松，资源用 Addressables / AssetBundle 按需下载 |
| URP Shader 在微信 WebGL 编译失败 | App 真机重新评估 URP Mobile 或 Built-in 移动端 Shader |
| 微信小游戏能力强绑定 | 平台能力抽象，可接微信、手机号、游客、广告、支付、多渠道 |

### 1.2 产品形态

| 项 | 内容 |
|---|---|
| 产品类型 | 体感拳击对战健身游戏 |
| 客户端 | Unity 2022.3 LTS 原生 App |
| 首发平台 | Android |
| 后续平台 | iOS |
| 核心卖点 | 真实出拳驱动角色、在线实时对战、地区排行、装扮收集 |
| 识别方式 | 手机前置摄像头 + 端侧人体关键点识别 |
| 推荐屏幕方向 | 首版建议竖屏，便于前置摄像头取景和单手操作；如沿用原横屏设计需重新验证摄像头取景范围 |
| 联网方式 | HTTPS + WebSocket |
| 商业化 | 激励广告、装扮付费、活动礼包、会员/训练计划可后续扩展 |

### 1.3 App 版目标

1. 将姿态识别从服务端主链路迁移到端侧，降低动作延迟和云 API 成本。
2. 复用当前 Unity 项目的游戏流程、战斗规则、动作判定、角色动画和服务端雏形。
3. 建立平台抽象层，使小游戏版和 App 版可以共存，不互相污染业务逻辑。
4. 首发 Android MVP，优先验证体感体验、真机性能、发热、耗电和联网对战。
5. Android 稳定后再扩展 iOS，避免双端原生插件同时开发带来的风险。

## 二、已有技术资产复用评估

### 2.1 可直接复用

| 模块 | 当前资产 | App 版处理 |
|---|---|---|
| Unity 项目结构 | `Assets/Scripts` 下已有 Core、Combat、PoseDetection、Network、UI、Character 等模块 | 保留 |
| 游戏流程 | 加载页、主菜单、对战、结算流程已跑通 | 保留并重做 UI 表现 |
| 战斗规则 | BO3、HP、AI 对手、出拳/防御/闪避事件 | 保留 |
| 动作判定 | `ActionDetector`、`PoseConfig`、`PoseFrame` 已围绕人体关键点设计 | 保留，输入源改为原生端侧姿态识别 |
| 服务端框架 | NestJS 服务端已有用户、房间、匹配、对战、排行、姿态接口雏形 | 保留并补全持久化和实时对战 |
| WebSocket 思路 | 已有网络同步桥接和原生 ws 网关方向 | 保留协议设计，App 端换成 Unity 原生 WebSocket 库 |
| 美术/动画规划 | 擂台、角色、动作、拳套、装扮规划 | 保留，按 App 性能重新设定规格 |

### 2.2 需要替换或重构

| 模块 | 当前实现 | App 版调整 |
|---|---|---|
| 姿态采集 | `Assets/Plugins/WebGL/PoseBridge.jslib` 调用微信 VKSession / wx camera | 替换为 Android/iOS 原生摄像头插件 |
| 姿态推理 | 服务端腾讯云/MediaPipe fallback | 主链路改为端侧 MediaPipe / MoveNet，服务端只保留兜底和风控 |
| 微信 SDK | `WechatBridge.jslib` + `wx.*` API | 改为微信 OpenSDK / 渠道 SDK，封装到平台层 |
| 网络桥接 | `NetworkBridge.jslib` 调 `wx.connectSocket` | 改为 Unity 原生 WebSocket 客户端 |
| 渲染管线 | 为微信 WebGL 兼容切到 Built-in RP | App 真机重新验证 URP Mobile 或保留 Built-in 移动端优化 |
| 包体资源 | 小游戏 CDN + WebGL 数据包 | App 首包 + Addressables / AssetBundle + CDN |
| 权限与隐私 | 微信小游戏权限弹窗 | Android/iOS 摄像头、网络、定位、广告标识符等权限合规 |

### 2.3 不建议复用为主链路

| 旧方案 | 不建议原因 | App 版替代 |
|---|---|---|
| 每帧上传摄像头图像到服务端识别 | 延迟高、成本高、隐私压力大、弱网体验差 | 端侧姿态识别 |
| 前端内置云 API 密钥 | 安全风险高 | 服务端签名和密钥托管 |
| 继续依赖 WebGL JS Bridge | App 中没有必要绕 WebGL | 原生插件 + Unity C# 接口 |
| 一开始 Android/iOS 双端并行 | 原生插件和审核链路会放大风险 | Android MVP 先跑通，再做 iOS |

### 3.2 核心数据流

#### 体感动作链路

```
前置摄像头
→ 原生帧预处理（旋转 / 镜像 / resize）
→ 端侧姿态识别模型
→ 33 点或 17 点人体关键点
→ Unity PoseBridge
→ ActionDetector
→ EventBus
→ CombatManager / Animator
→ NetworkCombatSync 上报动作事件
```

#### 在线对战链路

```
玩家本地识别动作
→ 客户端立即播放动作和预反馈
→ WebSocket 上报动作事件
→ 服务端校验动作时间窗、状态、冷却和命中
→ 服务端返回伤害结果
→ 双方客户端同步 HP、回合、胜负、表现效果
```

#### 资源更新链路

```
App 启动
→ 拉取远程版本配置
→ 检查 Addressables / AssetBundle 资源版本
→ 下载必要资源
→ 校验 hash
→ 进入游戏
```

## 四、客户端技术方案

### 4.1 Unity 层

| 模块 | 推荐方案 | 说明 |
|---|---|---|
| Unity 版本 | Unity 2022.3 LTS | 沿用当前项目，降低迁移成本 |
| 脚本后端 | IL2CPP | App 正式包使用 IL2CPP |
| 渲染管线 | URP Mobile 优先验证，Built-in 作为保底 | App 解除 WebGL Shader 限制，但仍需低端机真机测试 |
| 输入 | 触控 UI + 姿态动作 | 出拳可由姿态触发，按钮保留为调试和无摄像头兜底 |
| 动画 | Animator + Humanoid Rig | 继续复用动作资源 |
| 数据配置 | ScriptableObject + 远程 JSON | 本地默认配置，服务端可灰度下发阈值 |
| 网络 | NativeWebSocket / BestHTTP / 自研 WebSocket 封装 | 不再使用 `wx.connectSocket` |
| 资源 | Addressables | 角色、服装、拳套、特效可按需下载 |

### 4.2 平台抽象层

App 版应新增平台服务接口，避免业务逻辑直接依赖 Android、iOS 或微信 SDK。

```csharp
public interface IPlatformService
{
    void Login();
    void Share(string title, string imageUrl, string payload);
    void ShowRewardedAd(string placementId);
    void RequestCameraPermission();
    string GetDeviceLevel();
    string GetChannel();
}
```

推荐实现：

| 实现 | 用途 |
|---|---|
| `EditorPlatformService` | Unity Editor 调试 |
| `AndroidPlatformService` | Android 原生 App |
| `IOSPlatformService` | iOS 原生 App |
| `WechatMiniGamePlatformService` | 如需保留小游戏版本，可继续维护 |

### 4.3 姿态识别接口

Unity 上层不关心底层模型，只关心统一姿态帧。

```csharp
public interface IPoseProvider
{
    event System.Action<PoseFrame> OnPoseFrame;
    event System.Action<string> OnPoseError;
    void Start(PoseRuntimeOptions options);
    void Stop();
    bool IsRunning { get; }
}
```

推荐实现：

| 实现 | 输入源 | 用途 |
|---|---|---|
| `NativeMediaPipePoseProvider` | Android/iOS 原生 MediaPipe | App 主链路 |
| `NativeMoveNetPoseProvider` | TFLite MoveNet | 低端机或备选模型 |
| `ServerPoseProvider` | 服务端 `/pose/detect` | 兜底、调试、风控，不做主链路 |
| `MockPoseProvider` | 录制数据 / 模拟数据 | Editor 调试、自动化测试 |
| `WebGLPoseProvider` | 微信 JS Bridge | 保留小游戏版本时使用 |

## 五、端侧姿态识别方案

### 5.1 推荐主方案：MediaPipe Pose Landmarker

| 项 | 说明 |
|---|---|
| 模型 | MediaPipe Tasks Pose Landmarker |
| 输出 | 33 个关键点，与当前 `PoseFrame` 结构最匹配 |
| Android | MediaPipe Tasks Android + CameraX |
| iOS | MediaPipe Tasks iOS + AVFoundation |
| 优点 | 迁移成本低、点位完整、适合拳击动作判定 |
| 风险 | 低端机性能、发热、模型包体需真机验证 |

### 5.2 备选方案：MoveNet Lightning

| 项 | 说明 |
|---|---|
| 模型 | MoveNet Lightning / TFLite |
| 输出 | 17 个关键点 |
| 优点 | 推理速度快，适合低端机 |
| 缺点 | 点位少，需要映射到当前 `PoseFrame` 或调整判定逻辑 |
| 使用场景 | 低端 Android 机、性能档位降级 |

### 5.3 服务端姿态识别定位

App 版本不建议继续将服务端姿态识别作为主链路。服务端姿态接口可以保留，但定位调整为：

1. 开发调试：对比端侧模型输出，辅助调参。
2. 低端机兜底：端侧模型无法运行时，降帧上传关键帧。
3. 风控抽检：对异常高分、疑似作弊对局进行抽样验证。
4. 运营分析：采集非原始图像的动作统计，不保存用户视频。

### 5.4 原生插件职责

| 职责 | Android | iOS |
|---|---|---|
| 摄像头 | CameraX | AVFoundation |
| 图像方向 | 根据传感器方向旋转 | 根据设备方向旋转 |
| 前置镜像 | 原生层统一处理 | 原生层统一处理 |
| 推理线程 | 后台线程 / Executor | 后台队列 |
| 关键点输出 | JSON 或 Native Array | JSON 或 Native Array |
| 回调 Unity | UnitySendMessage / JNI | UnitySendMessage / C 接口 |
| 权限 | Android runtime permission | iOS permission prompt |

### 5.5 姿态数据协议

统一输出 MediaPipe 风格数据，即使底层使用 MoveNet，也在原生层或 Unity 适配层完成映射。

```json
{
  "timestamp": 1780000000,
  "source": "mediapipe",
  "latencyMs": 38,
  "imageWidth": 640,
  "imageHeight": 480,
  "mirror": true,
  "landmarks": [
    { "x": 0.51, "y": 0.32, "z": -0.02, "v": 0.96 }
  ]
}
```

### 5.6 识别性能目标

| 指标 | 低端机 | 中端机 | 高端机 |
|---|---|---|---|
| 游戏帧率 | 30 FPS | 45-60 FPS | 60 FPS |
| 姿态识别帧率 | 10-15 FPS | 15-24 FPS | 24-30 FPS |
| 姿态延迟 | ≤ 100ms | ≤ 70ms | ≤ 50ms |
| 单局发热 | 可接受，无明显降频 | 稳定 | 稳定 |
| 单局耗电 | 需实测控制 | 可接受 | 可接受 |

### 5.7 动作判定策略

1. 姿态识别只输出关键点，不直接决定游戏动作。
2. Unity `ActionDetector` 继续负责出拳、防御、闪避、轻重拳判定。
3. 开局做 3-5 秒校准，记录玩家中立姿态和身体尺度。
4. 动作阈值支持远程配置，便于按机型、模型、版本灰度调整。
5. 姿态帧过期时只更新姿态展示，不触发动作，避免网络或线程延迟造成误判。
6. 保留按钮调试模式，用于测试、无摄像头设备和审核演示。

## 六、对战系统方案

### 6.1 同步模式

App 版继续采用 **状态同步 + 服务端权威裁定**，不建议首版做帧同步。

| 决策 | 推荐 | 理由 |
|---|---|---|
| 同步对象 | 动作事件、HP、回合状态 | 拳击动作是离散事件，不需要逐帧同步骨骼 |
| 客户端表现 | 本地即时播放动作 | 保证手感 |
| 服务端职责 | 校验动作冷却、命中、伤害、胜负 | 控制作弊和结算可信度 |
| 网络协议 | WebSocket | 实现简单，移动端支持稳定 |
| 后期优化 | KCP / WebRTC 可选 | 需要更低延迟时再评估 |

### 6.2 客户端上行动作事件

```json
{
  "type": "PLAYER_ACTION",
  "matchId": "m_10001",
  "playerId": "p_1",
  "action": "right_straight",
  "power": "heavy",
  "clientTs": 1780000000,
  "poseConfidence": 0.91,
  "latencyMs": 42
}
```

### 6.3 服务端下行裁定结果

```json
{
  "type": "COMBAT_RESULT",
  "matchId": "m_10001",
  "attacker": "p_1",
  "target": "p_2",
  "action": "right_straight",
  "damage": 10,
  "targetHp": 72,
  "roundIndex": 1,
  "roundTimeLeft": 43.2
}
```

### 6.4 反作弊思路

| 风险 | 应对 |
|---|---|
| 客户端伪造高频出拳 | 服务端校验动作冷却、最大出拳频率、回合状态 |
| 修改伤害数值 | 伤害只由服务端计算，客户端只上报动作 |
| 模拟请求刷战绩 | 登录态、设备标识、签名、频率限制、战斗时长校验 |
| 姿态识别被跳过 | 排位模式要求姿态检测在线，按钮模式不计排行 |
| 异常高胜率 | 服务端记录对战日志，抽样复核 |

## 七、服务端技术方案

## 十一、包体与性能优化

### 11.1 包体目标

| 项 | 目标 |
|---|---|
| Android 首包 | 100-200MB 内优先 |
| iOS 首包 | 200MB 内优先，按 App Store 蜂窝下载限制调整 |
| 模型文件 | 首版只内置一个主模型，备选模型按需下载 |
| 可选资源 | 装扮、活动、高清特效走 CDN |

### 11.2 性能目标

| 指标 | 目标 |
|---|---|
| 低端机游戏帧率 | 30 FPS 稳定 |
| 中高端游戏帧率 | 60 FPS |
| 姿态识别帧率 | 15 FPS 以上，越高越好 |
| 出拳到游戏反馈 | 100ms 内 |
| 在线对战 RTT | 200ms 内可玩，100ms 内体验较好 |
| 冷启动 | 5-8 秒内进入首页 |
| 单局温升 | 需真机实测，不应快速触发明显降频 |

### 11.3 性能优化方向

1. 摄像头输入分辨率控制在 256-640 宽度区间，根据机型动态调整。
2. 姿态识别和 Unity 渲染线程分离，避免阻塞主线程。
3. 推理帧率不必等于游戏帧率，低端机可 10-15 FPS 推理，Unity 插值和平滑。
4. 角色、场景、粒子按设备性能分档。
5. 动作判定使用短窗口平滑，避免关键点抖动造成误判。
6. 对战页不加载商城、装扮大图等非必要资源。
7. Addressables 下载失败时提供重试和降级，不阻断核心玩法。

## 十二、隐私、权限与合规

### 12.1 权限清单

| 权限 | 用途 | 是否必需 |
|---|---|---|
| 摄像头 | 体感动作识别 | 必需 |
| 网络 | 登录、匹配、排行、资源下载 | 必需 |
| 定位 | 地区排行辅助定位 | 可选，必须允许手动选择地区 |
| 相册/存储 | 保存或分享战绩图 | 可选 |
| 广告标识符 | 广告归因与投放 | 视广告 SDK 要求 |
| 通知 | 活动、好友邀请、训练提醒 | 可选 |

### 12.2 隐私原则

1. 摄像头画面默认只在本机处理，不上传原始视频。
2. 排位模式可上传动作事件和关键统计，不上传完整图像。
3. 服务端姿态抽检必须取得用户协议授权，并尽量只上传关键点或低频抽样。
4. 隐私政策明确说明摄像头用途、数据处理方式、保存周期和第三方 SDK。
5. 用户拒绝摄像头权限时，可以进入非排行训练/按钮演示模式，但不能参加体感排位。

### 12.3 上架注意事项

| 平台 | 注意事项 |
|---|---|
| Android 应用市场 | 健身、摄像头、广告、支付、隐私合规需逐渠道适配 |
| iOS App Store | 摄像头用途描述、Apple 登录、IAP、健康/减肥宣传需谨慎 |
| 微信相关 | App 接微信登录/分享需申请移动应用，和小游戏 AppID 不是同一套流程 |

## 十三、开发阶段拆解

### 13.1 MVP 优先级

App 版建议分两条线推进：先做 Android MVP，验证体感体验；再补商业化和 iOS。

| 阶段 | 周期 | 目标 | 产出 |
|---|---|---|---|
| 1. 架构改造 | 5-7 天 | 抽象 PoseProvider、PlatformService、NetworkService | Unity 可同时支持 Editor Mock 和 App Native 输入 |
| 2. Android 姿态插件预研 | 7-10 天 | CameraX + MediaPipe 跑通，关键点回传 Unity | Android 真机 Demo |
| 3. 动作链路打通 | 7-10 天 | 姿态关键点 → ActionDetector → 角色动作 | 出拳、防御、闪避可用 |
| 4. App 渲染与性能调优 | 7-10 天 | URP/Built-in 真机对比，性能分档 | 稳定 30/60 FPS 配置 |
| 5. 在线对战联调 | 10-14 天 | WebSocket、匹配、房间、伤害裁定 | 可联网 1v1 |
| 6. App UI 与权限流程 | 7-10 天 | 登录、隐私、摄像头授权、校准页、设置页 | 完整首版流程 |
| 7. 资源体系 | 5-7 天 | Addressables、CDN、版本检查 | 可下载装扮资源 |
| 8. 商业化接入 | 7-10 天 | 微信登录/分享、激励广告、支付雏形 | 可运营版本 |
| 9. 测试与灰度 | 10-14 天 | 机型、发热、耗电、弱网、崩溃测试 | Android MVP 灰度包 |
| 10. iOS 适配 | 15-25 天 | AVFoundation + MediaPipe iOS + IAP + 审核 | iOS TestFlight 包 |

ML Kit Pose Detection（Google 官方安卓 SDK）或 TFLite + MoveNet 的方式做端侧识别


### 13.2 工期预估

| 范围 | 预估 |
|---|---|
| Android 技术验证版 | 20-30 天 |
| Android MVP 可灰度版 | 55-75 天 |
| Android + iOS 双端上线版 | 85-120 天 |

具体工期取决于原生姿态插件、UI 设计稿、美术资产、服务端持久化和商业化 SDK 的完成度。

## 十四、测试方案

### 14.1 机型测试矩阵

| 档位 | Android 建议 | iOS 建议 |
|---|---|---|
| 低端 | 3-4 年前中低端机，4GB 内存 | iPhone XR / iPhone 11 |
| 中端 | 主流骁龙 7 系 / 天玑 8 系 | iPhone 12 / 13 |
| 高端 | 骁龙 8 系 / 天玑 9 系 | iPhone 14 / 15 及以上 |

### 14.2 必测场景

1. 弱光、背光、普通室内光线下的姿态识别。
2. 玩家离手机 1 米、1.5 米、2 米时的关键点稳定性。
3. 左右直拳、上勾拳、防御、左右闪避、下蹲的误判率和漏判率。
4. 连续 3 局对战后的发热、掉帧、耗电。
5. 来电、切后台、锁屏、网络切换后的恢复。
6. 摄像头权限拒绝、再次授权、隐私协议未同意等边界流程。
7. WebSocket 断线重连、匹配取消、房间超时、战斗中掉线。
8. Addressables 下载失败、CDN 超时、资源版本不一致。

### 14.3 验收指标

| 指标 | 验收标准 |
|---|---|
| 姿态可用率 | 主流测试机型可稳定识别上半身 |
| 出拳响应 | 体感出拳到角色响应主观无明显拖沓 |
| 误判率 | 常规站姿晃动不频繁触发出拳 |
| 漏判率 | 标准直拳/上勾拳大部分可触发 |
| 对战稳定性 | 10 分钟内连续匹配/房间对战无严重断线 |
| 帧率 | 对战场景低端机 30 FPS，中高端 60 FPS 优先 |
| 崩溃率 | 灰度版本需重点跟踪，稳定后再扩大投放 |

## 十五、风险清单

| 风险 | 影响 | 应对 |
|---|---|---|
| 端侧姿态识别低端机性能不足 | 卡顿、发热、识别降帧 | MediaPipe/MoveNet 双模型、分辨率动态调整、性能档位 |
| 前置摄像头取景不稳定 | 玩家动作不完整，误判漏判 | 校准页引导站位，显示轮廓/关键点提示 |
| Android 机型碎片化 | 摄像头方向、镜像、权限差异 | 建立机型测试矩阵，原生层统一处理方向和镜像 |
| iOS 审核对摄像头和减肥宣传敏感 | 审核被拒 | 隐私文案清晰，避免夸大减肥效果，突出运动娱乐 |
| App 获客比小游戏重 | 下载转化低 | 保留微信分享、战绩图、房间邀请，后续考虑小游戏导流 App |
| 商业化 SDK 增加包体和合规风险 | 包体变大、隐私披露复杂 | 首版控制 SDK 数量，先接核心登录/分享/广告 |
| 服务端权威不足导致作弊 | 排行失真 | 服务端裁定伤害，记录动作频率和对战日志 |
| 热更新越界 | 平台审核风险 | 首版只做资源和配置热更新，不做代码热更 |

## 十六、从当前项目迁移到 App 的实施步骤

### 16.1 第一阶段：保留小游戏，新增 App 分支能力

1. 不删除 `Assets/Plugins/WebGL` 和微信小游戏代码。
2. 新增 `IPoseProvider`，将现有 WebGL 姿态桥接封装为一个 provider。
3. 新增 `NativePoseProvider` 占位，先用 Mock 数据跑通 Unity 层。
4. 新增 `IPlatformService`，把登录、分享、广告、权限从业务逻辑中抽离。
5. WebGL、小程序、Android、iOS 通过编译宏选择不同实现。

### 16.2 第二阶段：Android 原生姿态识别接入

1. 新建 Android Library 或 Unity Android Plugin。
2. 接入 CameraX，确认前置摄像头帧格式、方向、镜像。
3. 接入 MediaPipe Pose Landmarker，输出 33 点关键点。
4. 回传 Unity，复用当前 `PoseBridge.OnPoseData` 或改造为强类型回调。
5. 真机验证出拳、防御、闪避动作。

### 16.3 第三阶段：App 对战和服务端联调

1. Unity App 端接 WebSocket 客户端。
2. 服务端补全匹配、房间、战斗状态和结算。
3. 客户端本地即时播动作，服务端回包修正伤害和胜负。
4. 增加断线重连、战斗中切后台处理。

### 16.4 第四阶段：上架准备

1. 完成隐私政策、用户协议、第三方 SDK 清单。
2. 完成 Android 签名、渠道包、崩溃统计、日志上报。
3. 完成资源 CDN、版本配置、强更/热更策略。
4. 完成测试机型覆盖和灰度发布。
5. Android 稳定后启动 iOS 原生插件和 IAP 接入。

## 十七、推荐结论

App 版本推荐采用以下架构：

```
Unity 原生 App
+ Android/iOS 原生摄像头插件
+ MediaPipe Pose Landmarker 端侧姿态识别
+ MoveNet / TFLite 低端机备选
+ Unity C# 动作判定和战斗表现
+ WebSocket 服务端权威对战
+ NestJS + Redis + MySQL 后端
+ Addressables + CDN 资源体系
```

首发建议只做 **Android App MVP**。原因是当前项目最大的不确定性在端侧姿态识别的真机体验，Android 先跑通后，再投入 iOS 会更稳。

小游戏版本仍可保留为轻量传播入口，但真正追求体感拳击体验、画面表现和长期商业化时，App 版更适合作为主产品形态。
